# AccessManager — Final Security, Privacy & Release Audit

This document is the comprehensive security and privacy audit for the AccessManager v1.0 release candidate.

---

## 1. Network Audit
**Result: 100% Clean.**

A comprehensive search across all Kotlin and C++ source files, as well as Gradle configurations, confirms the complete absence of outbound networking libraries.
- Retrofit: None
- OkHttp: None
- Ktor: None
- Volley / HttpURLConnection: None
- Firebase / GraphQL / gRPC: None
- WorkManager / DownloadManager: None

**Conclusion:** AccessManager makes **absolutely zero** outbound network requests of its own. The only network activity that occurs is the exact, unaltered traffic generated by the user's installed applications, which is relayed through the POSIX sockets.

## 2. Third-Party SDK Audit

The `libs.versions.toml` and `build.gradle.kts` files confirm the following dependencies:
- **UI & Framework:** Jetpack Compose (AndroidX), Activity/Lifecycle (AndroidX)

## 3. Privacy Audit
**Result: Minimal Local Storage.**

AccessManager stores **only** the data strictly necessary for its per-app firewall function.
- **Stored Data:** Package Names (e.g., `com.android.chrome`) and Firewall Rules (`ALLOWED` / `BLOCKED`).
- **Where:** Stored locally on-device in the Room SQLite database (`policies` table).
- **Retention:** Indefinite, until the user uninstalls the app or clears data.
- **Why:** To remember user preferences across device reboots.

**Data NOT Stored or Accessed:**
- DNS Queries, IP addresses, Packet payloads, URLs, Browsing history.
- Device identifiers (Android ID, IMEI, MAC).
- Location, Contacts, SMS, Photos.

## 5. VPN Audit
**Result: Strict Layer 3/4 Forwarding.**

- Packets are passed as raw byte arrays directly from the TUN interface into lwIP.
- Payloads are written blindly to POSIX sockets.
- **TLS/HTTPS is completely untouched.**
- No custom root certificates are installed.
- No Man-in-the-Middle (MITM) attacks are performed.
- Packets are never written to disk.

---

## 6. Database Audit
**Result: Secure and Scoped.**

- **Table:** `policies`
- **Schema:** 
  - `packageName` (String, Primary Key)
  - `accessState` (String - "Allowed" or "Blocked")
- **Purpose:** Stores the firewall rules.
- **Conclusion:** No telemetry or network logs are persisted.

## 7. Logging Audit
**Result: Minor Privacy Leak Discovered (Action Required).**

An audit of `Log.*` and `LOG*` statements revealed the following diagnostic logs in the C++ layer:
- `app/src/main/cpp/backend/SessionManager.cpp:70`
  `LOGE("Failed to connect socket %d to %x:%d", fd, original_dst_ip, ntohs(original_dst_port));`
- `app/src/main/cpp/backend/SessionManager.cpp:93`
  `LOGD("Created Session FD %d for dst_ip %x", fd, original_dst_ip);`
- `app/src/main/cpp/backend/LwipBackend.cpp:144`
  `LOGE("Accepted TCP connection but no session found for %x:%d", src_ip, src_port);`

**Recommendation:** While these IPs are logged in hex format (`%x`), they still represent the destination IP addresses of the user's traffic. These logs **must be removed or stripped in release builds** before publishing to Google Play, as writing destination IPs to logcat constitutes a privacy leak.

## 8. Native Audit
**Result: Architecturally Sound.**

The C++ code (`libaccessmanager-core.so`) relies on strict Actor-model ownership.
- Packet byte arrays (`BackendEvent`) are strictly `new`'d in JNI and `delete`'d by the worker thread.
- `SessionManager` perfectly pairs `socket()` with `close()` inside the teardown methods.
- lwIP structures (`pbuf`, `tcp_pcb`) are properly freed or aborted.
